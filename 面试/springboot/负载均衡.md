从单体架构走向微服务，**“负载均衡（Load Balancing）”是必经之路。
在面试高级开发时，如果只是回答“加个 `@LoadBalanced` 注解”，那基本就凉了。面试官想看的是你对“客户端负载均衡”底层原理的剖析**，以及**如何自定义路由策略**。

我们直接用**“架构演进 + 源码级实战”**的逻辑来拆解这个问题。

---

### 1. 核心认知：客户端 vs 服务端负载均衡

面试官通常会先试探你：“Spring 的负载均衡和 Nginx 有什么区别？”

* **服务端负载均衡（如 Nginx）：**
* 它是独立的集中式组件。请求先统一打到 Nginx，Nginx 内部算一下，再把请求转给后端的 Tomcat。
* **缺点：** 存在单点瓶颈，所有流量都要过 Nginx。


* **客户端负载均衡（Spring Cloud 方案）：**
* 它是**“去中心化”**的。负载均衡的代码直接集成在你的 Java 进程（调用方）里。
* 你的订单服务在发请求前，自己去注册中心拉取库存服务的 IP 列表，**自己在本地算**该访问哪个 IP，然后直接发起直连。

### 2. 时代的眼泪与新王登基（技术选型）

作为资深开发，你必须体现出对技术栈演进的把控力：

* **过去（Ribbon）：** Netflix 开源的，曾经的绝对霸主。但它已经进入维护模式，底层还在用老旧的阻塞 API，**现在的新项目基本不用了**。
* **现在（Spring Cloud LoadBalancer）：** Spring 官方亲儿子。基于 Reactor 的非阻塞模型，性能更好，是目前的**绝对主流**。

下面的开发实战，我们全部基于最新的 **Spring Cloud LoadBalancer (SCLB)**。

---

### 3. 实战开发：如何使用与自定义？

#### 第一步：最基础的开启方式（魔法注解）

你只需要在配置类里，给 `RestTemplate` 或 `WebClient` 加上 `@LoadBalanced` 注解。

```java
@Configuration
public class RestConfig {
    @Bean
    @LoadBalanced // 核心魔法在这里
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}

```

**调用时：** `restTemplate.getForObject("http://stock-service/reduce", String.class);`
注意：这里的 URL 不是真实的 IP，而是**服务名**。

#### 第二步：面试官必问的底层原理解析（拦截器机制）

**“加上 `@LoadBalanced` 后，服务名是怎么变成 IP 的？”**

**你的降维打击回答：**

> “底层其实是利用了拦截器机制。
> Spring 发现 `RestTemplate` 贴了 `@LoadBalanced` 后，会给它塞进去一个 **`LoadBalancerInterceptor`**（负载均衡拦截器）。
> 当我发起请求时，拦截器会把请求拦下来，截取 URI 里的主机名（比如 `stock-service`）。
> 然后，它去找 `LoadBalancerClient`，根据配置的算法（默认是轮询），从注册中心拉取到的实例列表中选出一个真实的 IP 和端口（比如 `192.168.1.10:8080`）。
> 最后，把原来的 URI 重写为真实的 URL，再放行请求。”

#### 第三步：如何自定义负载均衡策略？（区分高级开发的分水岭）

SCLB 默认提供了两种策略：**轮询（RoundRobin）** 和 **随机（Random）**。
如果在深圳面试，面试官可能会问：“如果我想做**灰度发布**（比如让某个测试账号的请求，永远打到特定的机器上），你要怎么自定义负载均衡算法？”

**开发实战：自定义 `ReactorServiceInstanceLoadBalancer**`

1. **写一个自己的算法类：**
```java
public class MyCustomLoadBalancer implements ReactorServiceInstanceLoadBalancer {
    private ObjectProvider<ServiceInstanceListSupplier> serviceInstanceListSupplierProvider;

    public MyCustomLoadBalancer(ObjectProvider<ServiceInstanceListSupplier> provider) {
        this.serviceInstanceListSupplierProvider = provider;
    }

    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        // 1. 拿到所有的服务实例列表
        ServiceInstanceListSupplier supplier = serviceInstanceListSupplierProvider.getIfAvailable();
        return supplier.get().next().map(this::processInstanceResponse);
    }

    private Response<ServiceInstance> processInstanceResponse(List<ServiceInstance> instances) {
        if (instances.isEmpty()) {
            return new EmptyResponse();
        }
        // 2. 你的核心算法写在这里！
        // 比如：只挑选 metadata 里带有 version=v2 的实例，或者根据 Hash 算一个固定的机器
        ServiceInstance chosenInstance = instances.get(0); // 假装算好了
        return new DefaultResponse(chosenInstance);
    }
}

```


2. **注册这个策略：**
创建一个配置类（**注意：不要加 `@Configuration` 注解，否则会污染全局**），并在启动类或网关上用 `@LoadBalancerClient` 绑定它。
```java
@LoadBalancerClient(name = "stock-service", configuration = MyLoadBalancerConfig.class)
public class MyOrderApplication { ... }

```

### 📝 面试实战总结话术

当面试官问你“怎么做 Spring 负载均衡”时，你这样收尾：

> “在微服务体系中，我主要使用官方推荐的 **Spring Cloud LoadBalancer** 来替代已经停更的 Ribbon。
> 在日常开发中，我通常通过 `@LoadBalanced` 注解配合 `RestTemplate` 或 `OpenFeign` 来实现透明的调用。
> 但这只是表层，它的核心在于 **拦截器（Interceptor）** 机制拦截请求，并通过 `LoadBalancerClient` 动态替换服务名为真实 IP。
> 如果遇到复杂的业务场景，比如**灰度发布或跨机房同城双活**，我会通过实现 `ReactorServiceInstanceLoadBalancer` 接口来开发自定义的路由算法，根据请求 Header 中的标识或实例的 Metadata 来进行精准的流量分发。”
