### 🥉 青铜段位：最原始的 `SETNX`（全是坑）

> **话术：** “最早期，大家都是用 Redis 的 `SETNX` (Set if Not Exists) 命令来实现。”

1. **加锁：** `SETNX lock_key 1`。如果返回 1，说明没人用，我拿到了锁。
2. **解锁：** `DEL lock_key`。
3. **致命缺陷（死锁）：** 如果我刚拿到锁，我的 Java 进程突然崩溃了（或者服务器断电），`DEL` 没来得及执行。那么这个锁就**永远留在 Redis 里了**，其他所有服务器都在傻等，系统彻底死锁。

---

### 🥈 白银段位：引入过期时间（依然有坑）

> **话术：** “为了防止死锁，必须给锁加一个过期时间。但这里有个关于‘原子性’的坑。”

1. **错误示范：** 先 `SETNX`，再 `EXPIRE`。如果这两步中间断电了，依然死锁。
2. **正确做法：** 从 Redis 2.6.12 开始，`SET` 命令支持了复合参数。必须用一条命令保证原子性：
`SET lock_key unique_value NX PX 30000` (如果不存在则设置，并且 30 秒后自动过期)。
3. **致命缺陷（锁被别人误删）：**
* 线程 A 拿到锁（过期时间 10 秒）。
* A 的业务代码太复杂，跑了 15 秒！第 10 秒的时候，Redis 自动把锁释放了。
* 第 11 秒，线程 B 进来拿到了锁。
* 第 15 秒，线程 A 终于跑完了业务，执行了 `DEL lock_key`。**结果它把线程 B 的锁给删了！** 这会导致整个系统的锁机制彻底崩溃。



---

### 🥇 黄金段位：UUID 标识 + Lua 脚本原子解锁

> **话术：** “为了防止误删别人的锁，加锁和解锁都必须做严格的校验。”

1. **加锁时的改进：** 刚才提到的 `unique_value`，千万不能随便写个 1。必须用 **UUID + ThreadId**。
`SET lock_key "uuid-thread1" NX PX 30000`
2. **解锁时的改进（结合咱们刚才聊的 Lua）：**
解锁时，必须先 GET 判断一下这个锁到底是不是我的，如果是，才能 DEL。**查和删必须用 Lua 脚本保证原子性。**
```lua
if redis.call("get",KEYS[1]) == ARGV[1] then
    return redis.call("del",KEYS[1])
else
    return 0
end

```

### 🏆 王者段位：Redisson 与看门狗（生产标准答案）

> **话术：** “刚才说的虽然严谨，但依然没解决‘业务执行时间超过了锁过期时间’的问题。在真正的生产环境中，我们绝对不会手写这些逻辑，而是直接使用 **Redisson** 框架。”

向面试官抛出 Redisson 的核心机制：**Watch Dog（看门狗）**。

1. **加锁机制：** Redisson 底层完全封装了刚才说的 Lua 脚本，保证加锁的原子性，而且它存的数据结构不是简单的 String，而是 **Hash**（因为要支持可重入锁，记录当前线程重入的次数）。
2. **看门狗机制（核心原理）：**
* 当我们调用 `lock.lock()` 且不指定超时时间时，Redisson 默认会给锁设置 30 秒的有效期。
* 同时，它会在后台启动一个 **TimerTask 定时任务（看门狗）**。
* 这个看门狗每隔 **10 秒**（默认有效期的 1/3）就会去看一下：“哎，拿到锁的那个线程跑完了没有？”
* 如果业务还没跑完（线程还活着），看门狗就会向 Redis 发送一段 Lua 脚本，把这个锁的过期时间**重新续期（重置回 30 秒）**。


3. **完美闭环：** 只要你的业务没执行完，锁就永远不过期。如果你这台机器宕机了，看门狗也跟着死了，没人给锁续期，30 秒后锁自动释放，不会死锁。
