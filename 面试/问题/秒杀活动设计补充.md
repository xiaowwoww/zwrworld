老周，这是一道**系统设计（System Design）**的大题。

在面试高级开发或架构师时，面试官不是让你写代码，而是让你画**架构图**。

秒杀系统的核心设计理念只有两个字：**“漏斗”**。
我们要把巨大的流量（比如 100万 QPS）一层一层地挡在外面，最后只放极少量的请求（比如 100 QPS）进入数据库。

以下是为你定制的**《秒杀系统全链路设计方案》**，分为**流量拦截、缓存扣减、异步下单、数据库兜底**四个阶段。

---

### 🏛️ 整体架构图（漏斗模型）

1. **用户端（APP/H5）：** 按钮置灰，CDN 静态资源。
2. **网关层（Nginx/Gateway）：** 限流、黑名单、动静分离。
3. **服务层（Java）：** 本地缓存标记、Redis 预减库存。
4. **消息层（MQ）：** 削峰填谷。
5. **数据层（DB）：** 最终持久化。

---

### ⚔️ 第一阶段：活动开始前（预热与隔离）

不要等活动开始了才去查数据库，那时候数据库必死无疑。

1. **服务隔离（Deployment）：**
* 秒杀服务必须**独立部署**。千万别和主业务（如下单、支付）混在一起，万一秒杀挂了，别把公司主站拖垮。
* 数据库也要独立（或分表）。


2. **页面静态化（CDN）：**
* 秒杀页面（HTML/CSS/JS）提前推送到 **CDN** 节点。用户访问时，不需要回源到你的服务器，减轻带宽压力。
* **按钮控制：** 还没到 10:00，前端按钮强制置灰（Disabled），或者倒计时。


3. **库存预热（Cache Pre-warm）：**
* 活动开始前 5 分钟，用定时任务把数据库里的 100 个库存，加载到 Redis 中。
* `SET seckill_stock:1001 100`



---

### ⚔️ 第二阶段：活动进行中（流量拦截与扣减）

这是战斗最激烈的地方。我们要把 99.9% 的请求拦截在这里。

1. **网关限流（Rate Limiting）：**
* 在 Nginx 或 Spring Cloud Gateway 层，配置**漏桶算法**或**令牌桶算法**。
* 比如：限制单 IP 每秒只能访问 5 次。超过的直接返回“哎呀，太挤了”。


2. **防刷风控（Anti-Bot）：**
* **隐藏秒杀地址：** 不要直接暴露 `/seckill/buy` 接口。
* **动态 URL：** 用户点击“抢购”时，先请求一个 `/get_token` 接口，拿到一个随机的 Token（比如 MD5 签名）。
* 真正的下单接口是 `/seckill/buy?token=xyz...`。如果没有 Token，直接拒绝。


3. **本地缓存标记（Guava/Caffeine）—— （高级亮点）**
* Redis 也是有瓶颈的（单机几万 QPS）。
* 如果库存卖完了，Redis 里的库存是 0。
* 我们可以在 Java 内存里设置一个 `boolean isStockEmpty = false`。
* 一旦 Redis 返回 0，把这个标记设为 `true`。
* 后面的请求连 Redis 都不用查，直接在 Java 内存里就拦截掉了：“对不起，卖光了”。


4. **Redis 预减库存（Lua 脚本）：**
* 这就是我们刚才讨论的 Lua 脚本。
* **原子性**扣减：`if stock > 0 then decr; return true; else return false;`
* **每人限购 1 个：** 在 Lua 里同时检查 `SISMEMBER`（Set 集合），如果用户 ID 已经在集合里了，说明买过了，直接返回。



---

### ⚔️ 第三阶段：异步下单（削峰填谷）

Redis 扣减成功了，代表用户**“抢到了资格”**，但还没真正生成订单。

1. **发送 MQ 消息：**
* Java 服务收到 Redis 的“成功”信号后，组装一个消息（包含 UserID, GoodsID），扔进 **RocketMQ** 或 **Kafka**。
* 然后立即给前端返回：**“排队中，请稍候...”**（注意：此时不返回“成功”，而是“排队中”）。


2. **前端轮询（Polling）：**
* 前端每隔 2 秒调用一次 `/check_order_status` 接口，查询订单是否真的生成了。



---

### ⚔️ 第四阶段：数据库持久化（最终兜底）

消费者（Consumer）慢慢地从 MQ 里拉取消息，慢慢地写数据库。

1. **扣减真实库存：**
* 虽然 Redis 减了，但数据库也要减。
* **防超卖 SQL（乐观锁）：**
```sql
UPDATE seckill_goods 
SET stock_count = stock_count - 1 
WHERE goods_id = 1001 AND stock_count > 0;

```


* *注意：这里其实是双重保险。即使 Redis 放进来多余的请求，数据库这行 SQL 也能保证最后一道防线。*


2. **生成订单与幂等性：**
* 在 `t_order` 表插入订单记录。
* **唯一索引：** 给 `(user_id, goods_id)` 加唯一索引。万一 MQ 重复消费了，数据库会报错，保证一个用户真的只生成一笔订单。


3. **处理 Redis 与 DB 不一致（补偿）：**
* 如果 DB 扣减失败了（比如网络挂了），需要发消息回滚 Redis 的库存。

### 📝 35 岁架构师的“总结陈词”

> **面试官：** “这套方案有什么优缺点？”
> **你：**
> “这套方案是经典的**‘Redis + MQ + DB’**的三级火箭模式。
> **优点是：**
> 1. **抗并发：** 流量被 Nginx、本地缓存、Redis 层层过滤，到底层 DB 时已经很平缓了。
> 2. **高可用：** 异步下单，用户体验好，不会因为 DB 锁死而页面卡顿。
> 
> 
> **缺点是：**
> 1. **系统复杂：** 引入了 MQ、Redis、定时任务，维护成本高。
> 2. **数据一致性挑战：** 极端情况下（如 Redis 挂了但数据没落库），可能需要人工对账或脚本补偿。
> 
> 
> **但在 10W QPS 的场景下，这是为了保住系统稳定性的必要妥协。**”
