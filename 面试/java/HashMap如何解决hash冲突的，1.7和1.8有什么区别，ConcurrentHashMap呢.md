### 第一部分：HashMap 是如何解决 Hash 冲突的？

核心策略只有四个字：**拉链法 (Chaining)**。

1. **定位：** 拿到 Key，算出 Hash 值，通过 `(n - 1) & hash` 快速定位到数组的下标（Bucket）。
2. **冲突：** 如果这个下标已经有人了（Hash 冲突），这就叫“撞车”。
3. **解决：** 没关系，我在这个下标位置挂一个**链表**。新来的元素就像挂腊肠一样，挂在上面。

---

### 第二部分：HashMap 1.7 与 1.8 的生死对决

这是面试的**重灾区**，必须讲出“为什么改”。

#### 1. JDK 1.7 —— “数组 + 链表” (简单粗暴)

* **结构：** 纯粹的 `Entry` 数组 + 链表。
* **插入方式：** **头插法 (Head Insertion)**。
* *逻辑：*哪怕链表里有人了，新来的元素插在**头部**（最上面）。
* *设计初衷：* 作者觉得后插入的数据被访问的概率更大（热点数据），放在头部查得快。


* **致命死穴（面试必杀技）：**
* 在**多线程并发扩容（Resize）**时，因为头插法会改变链表的顺序（逆序），容易导致**链表成环（Infinite Loop）**。一旦成环，下次 `get()` 的时候 CPU 直接飙到 100%。
* *这也是为什么严禁在多线程下用 HashMap 的核心原因。*



#### 2. JDK 1.8 —— “数组 + 链表 + 红黑树” (性能怪兽)

* **结构升级：**
* 当链表长度 **> 8** 且数组长度 **>= 64** 时，链表会强制转为 **红黑树 (Red-Black Tree)**。
* 当红黑树节点数 **< 6** 时，退化回链表。


* **插入方式：** **尾插法 (Tail Insertion)**。
* *逻辑：* 新来的元素乖乖排在队尾。
* *解决痛点：* **扩容时保持链表顺序不变**，彻底解决了 1.7 中的“链表成环”死锁问题。


* **为什么引入红黑树？（降维打击）**
* 在 1.7 中，如果黑客故意构造一堆 Hash 冲突的 Key（Hash DoS 攻击），链表会变得巨长。查询复杂度从  退化成 ****，服务器直接卡死。
* 引入红黑树后，最差情况也就是 ****。这是**性能兜底**的策略。

### 第三部分：ConcurrentHashMap (CHM) —— 并发之王

HashMap 是线程不安全的，CHM 就是为了解决这个问题。它的演进更是**“锁粒度”**的教科书。

#### 1. JDK 1.7 CHM —— “分段锁” (Segment)

* **核心思想：** **化整为零**。
* **结构：** 它不锁整个大数组，而是把大数组切分成 16 个小段（Segment）。每个 Segment 本质上就是一个小的 Hashtable，继承自 `ReentrantLock`。
* **并发度：** 默认 16。也就是说，最多允许 16 个线程同时写。
* **缺点：** 锁的粒度还是有点粗。而且数据结构复杂（Segment -> HashEntry -> Chain），内存占用高。

#### 2. JDK 1.8 CHM —— “CAS + Synchronized” (原子为王)

* **核心思想：** **放弃分段，直捣黄龙（锁节点）**。
* **结构：** 跟 HashMap 1.8 一样（数组+链表+红黑树）。**彻底抛弃了 Segment**。
* **如何保证线程安全？（重中之重）**
1. **如果槽位是空的：** 使用 **CAS (Unsafe.compareAndSwapObject)** 尝试直接插入。**无锁，速度极快。**
2. **如果槽位有人（Hash冲突）：** 使用 **`synchronized`** 锁住当前槽位的**头节点 (Node)**。


* **优势：**
* **锁粒度细到了极致：** 只有发生 Hash 冲突的那两个线程才会竞争锁，其他不冲突的线程完全并发执行。
* **为什么用 synchronized 不用 ReentrantLock？** 因为 JDK 1.6 以后 synchronized 做了锁升级优化（偏向锁/轻量级锁），在低竞争下性能极好，而且比 ReentrantLock 省内存（不用每个节点都 new 一个 Lock 对象）。

> **面试官：** “说说 HashMap 和 ConcurrentHashMap 的区别及演进？”
> **你：**
> “HashMap 的核心是用**拉链法**解决冲突。
> **在 1.7 到 1.8 的演进中，最大的变化是两点：**
> 第一，数据结构引入了**红黑树**，把最差情况下的查询效率从  提升到了 ，防止 Hash 碰撞攻击。
> 第二，插入方式从**头插法**改为了**尾插法**，解决了并发扩容下的**死循环**问题。
> **至于 ConcurrentHashMap，它的演进本质是‘锁粒度的细化’：**
> 1.7 采用**分段锁（Segment）**，虽然比 Hashtable 全局锁好，但并发度仍受限于 Segment 个数。
> 而 1.8 **抛弃了 Segment**，直接利用 **CAS + synchronized** 锁住链表头节点。这做到了**‘能无锁就无锁（CAS），必须锁就锁最小单位（Node）’**，将并发性能推到了极限。”
