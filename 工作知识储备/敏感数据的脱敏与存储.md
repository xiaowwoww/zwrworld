在建行这种金融国家队里，一旦敏感数据（客户姓名、身份证、手机号、银行卡号、征信报告）发生明文泄露，这不叫 Bug，这叫**“重大安全事故”**，严重的是要直接移交合规甚至经侦部门的。

咱们 Java 后端处理敏感数据，其实就是在守住三大关口：**落库（怎么存）、打印（怎么记）、下发（怎么传）**。我用建行最典型的实战架构，给你把这三关的底层做法拆解清楚：

### 第一关：存储加密（落库）—— 绝对的“暗箱操作”

在银行的数据库里，敏感字段绝对不允许以明文形态存在。

* **老兵铁律：全面拥抱“国密算法（SM）”**
  在以前的互联网公司，你可能习惯用 AES 加密、MD5 摘要。但在建行，现在推行的是极其严格的**信创标准**。
* **对称加密：必须用 SM4。** 用于加密手机号、身份证号，保证存进去是密文，取出来能解密。
* **不可逆哈希：必须用 SM3。** 用于密码等不需要解密的字段。


* **架构师的终极难题：加密了怎么做 `WHERE` 查询？**
* **痛点：** 每次用 SM4 加密同一个手机号，如果带了随机盐（Salt），生成的密文是完全不一样的。那你怎么在数据库里执行 `SELECT * FROM t_user WHERE phone = '13812345678'`？
* **建行级实战打法（盲排索引 Blind Index）：**
  表结构里，一个敏感字段通常要拆成两个字段来存：
1. `phone_cipher`（手机号密文）：用 SM4 加密，用于数据还原。
2. `phone_hash`（手机号哈希检索列）：用 SM3（或加固定盐的 SHA-256）进行不可逆哈希。
   **查询逻辑：** 当你要按手机号查人时，Java 代码先用同样的 SM3 算法把前端传来的手机号算出 Hash 值，然后去查 `WHERE phone_hash = '那个Hash值'`。完美解决加密检索问题！





### 第二关：日志脱敏（打印）—— 防御“内鬼”的最强防线

这是刚进银行的外包兄弟最容易被开除的雷区。你可能只是为了排查 Bug，随手写了一句 `log.info("当前申请贷款的客户信息: {}", customerEntity);`，结果把包含明文身份证的 JSON 直接打到了生产服务器的日志文件里。

* **实战做法：拦截器与重写 `toString()**`
  建行的基础架构组一定提供了一个统一的**脱敏日志组件**。它的底层原理通常有两种，你入场后注意观察项目里用的是哪种：
1. **Logback/Log4j2 层的正则替换（全局拦截）：**
   在日志框架底层写一个自定义的 `MessageConverter`。只要探测到打印的字符串里有连续的 11 位数字（疑似手机号）或者 18 位数字（疑似身份证），直接用正则表达式替换成 `138****5678`。
2. **优雅的注解驱动（推荐）：**
   在你写的 Entity 或 DTO 类上，重写 `toString()` 方法。引入类似 fastjson/Jackson 的脱敏特性。
```java
public class CustomerInfo {
    @Desensitized(type = DesensitizeType.PHONE) // 行内自定义注解
    private String phone;
}

```


这样无论你怎么 `log.info`，打印出来的永远是打码后的字符串。



### 第三关：接口下发脱敏（传输）—— 蒙上前端的眼睛

前端页面（比如建行 App）需要展示客户资料时，除非客户点击了“闭眼”图标要求明文显示，否则默认下发的数据必须是脱敏的。

* **绝对红线：** 严禁后端把明文发给前端，让前端用 Vue/React 去做打码（比如截取字符串换成星号）。因为只要被黑客抓个包，明文数据就全漏了！脱敏必须在 Java 后端完成。
* **实战做法：Jackson 自定义序列化器**
  在 Spring Boot 给前端返回 JSON 的那一瞬间进行拦截替换。
```java
public class UserVO {
    private String name;

    // 在 JSON 序列化时，自动将 "张三" 变成 "张*"，"欧阳锋" 变成 "欧阳*"
    @JsonSerialize(using = ChineseNameDesensitizeSerializer.class)
    private String idCardName;
}

```


你只需要在对应的 VO（View Object）类上贴好这些注解，Spring MVC 就会在响应返回前，自动把敏感字段“打码”，安全、优雅且无侵入。
