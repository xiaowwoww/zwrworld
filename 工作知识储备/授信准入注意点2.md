### 1. 外部数据的“合法白嫖”：严格的缓存复用机制

我们昨天说，查一次人行征信可能要花 2 块钱，查外部反欺诈分要花 5 毛钱。

* **业务痛点：** 客户今天点申请，因为资料没填对被拒了；明天他又来点申请。如果你又去调一次外部接口，银行的钱就被你白白烧掉了。
* **老兵打法（有效期复用）：** 风控调外部接口前，第一步永远是查本地数据库的**“历史快照表”**或 Redis 缓存。
* 比如业务规定：人行征信的有效期是 30 天，反欺诈分的有效期是 7 天。
* 代码逻辑：先拿 `客户身份证号 + 业务场景` 去查最近一次的调用记录。如果在这个有效期内，直接把历史的 JSON 报文捞出来，反序列化后作为 Fact 喂给规则引擎。


* **🚫 致命红线（合规审查）：** 存在 Redis 里的敏感数据（如征信明细），**绝对不能是明文！** 必须用国密算法（如 SM4）加密后存入。被内审部门查出缓存库里有明文征信，是极其严重的违规事件。

### 2. 挡住羊毛党与黑产：极致的防刷与幂等控制

风控准入接口不仅是防风险的，它自己也是黑产（黑客）疯狂攻击的目标。他们会用脚本在一秒内发送上百次申请，试图寻找系统的并发漏洞。

* **常规防御：** 昨天提过的 `SETNX` 分布式锁。
* **风控专属防御（滑动窗口限流）：** 在网关层或代码的最前端，必须引入基于 Redis 的限流器（Rate Limiter）。
* **策略：** 限制同一个身份证号，1 分钟内最多只能调用 1 次准入接口；1 天内最多只能被拒绝 3 次。一旦超过阈值，系统连并发编排都不走，直接 `Fast-Fail` 抛出：“您的操作过于频繁，请明日再试”。这既保护了系统，又省了外调费用。



### 3. 不执着于“即时结果”：长流程的异步解耦（转人工信审）

老周，在规则引擎的判决里，除了“通过”和“拒绝”，往往还隐藏着第三个极其高频的状态：**“转人工（PENDING_MANUAL_REVIEW）”**。

* **业务场景：** 机器拿不准这个客户，需要分发给建行后台的信审员，人工打电话去核实。这个过程可能要等 2 个小时甚至 1 天。
* **架构解法：** 你的准入接口绝对不能卡着线程等人工审批！
* **第一步：** 你的代码接到规则引擎“转人工”的判定后，将这笔订单状态置为 `审批中`，并立刻返回给前端，释放 Tomcat 线程。
* **第二步：** 把这笔申请的资料打包，发送到 MQ（比如 Kafka 或 RocketMQ）中。
* **第三步：** 信审系统（另一个独立的微服务）消费这条消息，分配给具体的工作人员。
* **第四步：** 信审员在页面上点完“同意”，信审系统再发一条 MQ 消息出来，你的系统监听到后，再去推进下一步（比如放款）。
* **修心哲学：** 这种异步解耦，极其考验你“不执着于当下的闭环”的定力。把任务交出去，做好状态机的流转，等待回调即可。



### 4. 黄金自保法则：全链路 TraceID（MDC 追踪）

微服务时代，排查 Bug 就像在迷宫里找人。一个贷款申请，可能要经过网关、风控微服务、计息微服务、核心账务系统。

* **恐怖场景：** 业务群里发来一个截图：“老周，这个客户在 App 上报错了，赶紧看看”。你打开 Kibana（日志中心），发现全是密密麻麻的并发日志，根本不知道哪一行是这个客户的。
* **老兵标准动作：** 在项目代码里，确保每一次 HTTP 请求进来时，都在拦截器里生成一个全局唯一的 `TraceID`，并放入 SLF4J 的 `MDC (Mapped Diagnostic Context)` 中。
* 只要你在风控代码里打印 `log.info("开始处理准入...")`，日志框架会自动带上这个 TraceID：`[TraceID: 8a9b2c... ] 开始处理准入...`。
* 以后出问题，直接在日志系统里搜这个客户的手机号拿到 TraceID，然后用 TraceID 一搜，整个微服务链路里关于这个客户的所有动作一览无余，自证清白只需 5 秒钟。
