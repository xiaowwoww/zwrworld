# 一、业务上必须死记的红线（银行最看重）
1. **授信准入 = 能不能贷、贷多少、什么利率、什么期限**
   你写的每一行代码，都直接决定**能不能放款、放多少**。
   → 原则：**宁可拦截错，不能放过错。**

2. **风控规则绝对不能乱改**
    - 规则配置、阈值、白名单、黑名单、模型分
    - 没经过**业务+风控+测试+运维**确认，绝对不能动
    - 银行系统：**改动必留痕，上线必评审**

3. **数据一定要准**
   征信、人行数据、三方数据、内部评级、客户信息
   → 空值、异常值、负数、越界，必须全部拦截。

4. **一定要懂“拒绝码/拒绝原因”**
   每一笔拒绝都要有**标准编码**，方便审计、监管、回溯。
   你写代码时，**拒绝码不能乱写、不能漏**。

# 二、技术上你必须注意的点（Java 开发必避坑）
1. **幂等！幂等！幂等！**
   重试、回调、重发请求非常常见
   → 接口必须做**幂等**（订单号/申请号唯一）

2. **分布式事务一定要小心**
   银行多数据源、分库分表、分布式数据库很常见
   → 不要自己硬写事务，用**最终一致性、状态机、补偿机制**

3. **所有关键操作必须留日志**
    - 入参
    - 出参
    - 规则结果
    - 决策依据
      监管检查、审计、客诉，全靠日志。

4. **不能出现脏数据、漏更新、错更新**
   授信额度、状态、风控结果，一旦写错就是**生产事故**。
   → 写 SQL 先 WHERE，再 UPDATE，一定要加条件。

5. **接口必须做校验**
    - 非空校验
    - 长度校验
    - 格式校验
    - 范围校验
      银行系统：**外部进来的全是不可信数据**。

4. **版本、配置、脚本一定要规范**
