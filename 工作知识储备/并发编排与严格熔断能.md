### 第一步：给外部接口套上“防弹衣”（Resilience4j 熔断与限时）

在调用外部系统（比如人行征信、公安接口）的 Service 层，我们绝对不能直接发起 HTTP 请求，必须加上 `@CircuitBreaker`（熔断器）和 `@TimeLimiter`（时间限制）。

```java
@Service
@Slf4j
public class ExternalRiskService {

    // 1. 调用人行征信接口（极易超时和不稳定）
    // 配置名为 "pbocClient" 的熔断策略，如果超时或报错，自动进入 fallback 方法
    @TimeLimiter(name = "pbocClient")
    @CircuitBreaker(name = "pbocClient", fallbackMethod = "pbocFallback")
    public CreditReport checkPbocCredit(String idCard) {
        log.info("开始调用人行征信接口...");
        // 模拟外部 HTTP 调用
        return restTemplate.postForObject("http://pboc.gov/api/credit", idCard, CreditReport.class);
    }

    // 🚨 兜底逻辑（Fallback）：外部接口挂了怎么办？
    // 在风控里，征信挂了通常不能放行，而是返回一个特殊的“转人工”或“系统异常”状态
    public CreditReport pbocFallback(String idCard, Throwable t) {
        log.error("人行征信接口调用失败或触发熔断，身份证: {}", idCard, t);
        CreditReport fallbackReport = new CreditReport();
        fallbackReport.setStatus("UNKNOWN"); // 标记为未知，转由人工复核或稍后重试
        return fallbackReport;
    }
    
    // 2. 调用内部反欺诈引擎（同理）
    @CircuitBreaker(name = "antiFraudClient", fallbackMethod = "fraudFallback")
    public FraudResult checkAntiFraud(String idCard) {
        // ... 调用逻辑
    }
}

```

### 第二步：主战场的“并发编排”（CompletableFuture 实战）

到了你负责的“准入大网关”入口，我们要把上面的服务全部并发拉起来。

**银行老兵的绝对红线：** 严禁使用 `CompletableFuture` 默认的 `ForkJoinPool.commonPool()`！如果全行都用默认线程池，一个接口卡死，全行业务跟着崩溃。**必须自定义业务专属的线程池。**

```java
@Service
@Slf4j
public class RiskAdmissionFacade {

    @Autowired
    private ExternalRiskService externalRiskService;

    // 注入我们为风控准入专门隔离出来的自定义线程池
    @Autowired
    @Qualifier("riskAdmissionThreadPool")
    private Executor riskExecutor;

    public AdmissionResult executeAdmission(String idCard) {
        long startTime = System.currentTimeMillis();

        // 1. 并发发起三个外部调用（互不阻塞）
        CompletableFuture<CreditReport> pbocFuture = CompletableFuture.supplyAsync(
                () -> externalRiskService.checkPbocCredit(idCard), riskExecutor);

        CompletableFuture<FraudResult> fraudFuture = CompletableFuture.supplyAsync(
                () -> externalRiskService.checkAntiFraud(idCard), riskExecutor);

        CompletableFuture<PoliceResult> policeFuture = CompletableFuture.supplyAsync(
                () -> externalRiskService.checkPoliceStatus(idCard), riskExecutor);

        // 2. 编排：等待所有任务完成
        CompletableFuture<Void> allFutures = CompletableFuture.allOf(pbocFuture, fraudFuture, policeFuture);

        try {
            // 🚨 终极防线：给整个编排过程加上全局超时时间！
            // 比如规定：整个风控准入最多只能等 3 秒，等不到直接强行中断
            allFutures.get(3, TimeUnit.SECONDS);

            // 3. 收集结果（此时 get() 瞬间返回，因为上面已经等完了）
            CreditReport pbocReport = pbocFuture.get();
            FraudResult fraudResult = fraudFuture.get();
            PoliceResult policeResult = policeFuture.get();

            // 4. 将收集到的数据组装成 Fact，丢给规则引擎去判断
            return ruleEngine.evaluate(pbocReport, fraudResult, policeResult);

        } catch (TimeoutException e) {
            log.error("全局风控准入超时！耗时: {} ms", System.currentTimeMillis() - startTime);
            // 超时降级策略：打回申请，提示用户网络拥挤
            return AdmissionResult.reject("R999", "系统处理超时，请稍后再试");
        } catch (Exception e) {
            log.error("风控准入发生未知异常", e);
            return AdmissionResult.reject("R500", "系统内部错误");
        }
    }
}

```

### 1. 为什么要“快速失败（Fast-Fail）”？（省钱才是硬道理）

假设你的准入接口要查 3 个东西：

1. **行内黑名单（内部数据库）：** 耗时 20 毫秒，免费。
2. **公安身份核验（外部接口）：** 耗时 500 毫秒，调一次收费 0.5 元。
3. **人行征信（外部接口）：** 耗时 2 秒，调一次收费 2 元。

**小白的写法（`allOf` 死等）：**
三个任务并发跑。行内黑名单 20 毫秒就返回了“该客户是高危欺诈分子，拒绝”。但是，小白的代码还在傻傻地等公安和人行接口跑完。
**后果：** 客户明明已经被内部拉黑了，银行却还要白白浪费 2.5 秒的线程等待时间，并且白白扔给外部机构 2.5 块钱的查询费。如果一天有 10 万个这种黑户来申请，银行一天就白亏 25 万！

**架构师的写法（Fast-Fail 编排）：**
只要行内黑名单返回了“拒绝”，**立刻、马上掐断（Cancel）**正在排队或正在执行的公安和人行查询，直接给前端返回结果。

### 2. 核心实战代码：如何用 CompletableFuture 实现“一票否决”

Java 原生的 `CompletableFuture.allOf()` 有个致命缺陷：它必须等所有任务都执行完（或者抛出异常），不能做到“只要有一个结果是不符合预期的，就立刻中止其他所有任务”。

在建行的底层架构里，通常会有一套封装好的高阶玩法。下面我用纯 Java 8 代码给你还原这个“一票否决”的底层逻辑：

```java
@Service
@Slf4j
public class AdvancedRiskFacade {

    @Autowired
    @Qualifier("riskAdmissionThreadPool")
    private Executor riskExecutor;

    public AdmissionResult executeFastFailAdmission(String idCard) {
        // 1. 创建一个作为“总控”的 Future
        // 任何一个子任务如果判定“拒绝”，就会立刻让这个总控 Future 完成
        CompletableFuture<AdmissionResult> globalResult = new CompletableFuture<>();

        // 2. 发起：行内黑名单查询（极快）
        CompletableFuture<Void> internalTask = CompletableFuture.runAsync(() -> {
            boolean isBlack = checkInternalBlacklist(idCard);
            if (isBlack) {
                log.warn("命中行内黑名单，立刻触发全局快速失败！");
                // 核心：直接让总控结果“完成”，带上拒绝原因
                globalResult.complete(AdmissionResult.reject("R001", "命中行内黑名单"));
            }
        }, riskExecutor);

        // 3. 发起：人行征信查询（极慢且极贵）
        CompletableFuture<Void> pbocTask = CompletableFuture.runAsync(() -> {
            boolean isPbocBad = checkPboc(idCard);
            if (isPbocBad) {
                log.warn("人行征信不良，立刻触发全局快速失败！");
                globalResult.complete(AdmissionResult.reject("R002", "征信不良"));
            }
        }, riskExecutor);

        // 4. 将所有子任务收集起来，用于正常的“全部通过”场景
        CompletableFuture<Void> allTasks = CompletableFuture.allOf(internalTask, pbocTask);

        // 5. 如果所有任务都正常跑完，且没有触发上面的 globalResult.complete
        allTasks.thenRun(() -> {
            if (!globalResult.isDone()) {
                log.info("所有风控校验通过！");
                globalResult.complete(AdmissionResult.approve("放行"));
            }
        });

        try {
            // 6. 见证奇迹的时刻：主线程只在这个总控 Future 上等待
            // 只要任何一个子任务触发了 globalResult.complete，这里会瞬间返回！
            AdmissionResult finalResult = globalResult.get(3, TimeUnit.SECONDS);

            // 7. 优雅收尾（省钱动作）：只要总控拿到结果了，不管三七二十一，把还在跑的子任务全部 Cancel 掉！
            if (finalResult.isRejected()) {
                log.info("准入被拒绝，正在尝试取消其他未完成的昂贵查询...");
                pbocTask.cancel(true); // 给底层的 HTTP 线程发送中断信号
            }

            return finalResult;

        } catch (TimeoutException e) {
            log.error("风控全局超时");
            return AdmissionResult.reject("R999", "系统超时");
        } catch (Exception e) {
            log.error("风控未知异常", e);
            return AdmissionResult.reject("R500", "系统内部异常");
        }
    }

    // 模拟外部调用
    private boolean checkInternalBlacklist(String idCard) { /* 耗时 20ms */ return true; }
    private boolean checkPboc(String idCard) { /* 耗时 2000ms */ return false; }
}

```

### 3. 代码里的“神级细节”（老兵视角）

看懂上面这段代码，你需要体会其中三个极其精妙的细节：

* **自造“总控开关”：** 我们没有死等 `allTasks.get()`，而是 `new` 了一个空的 `globalResult`。这是控制流转的神来之笔。谁先查出问题，谁就有权按下这个开关（调用 `complete`）。
* **线程中断信号（`cancel(true)`）：** 当黑名单瞬间查出问题后，我们调用了 `pbocTask.cancel(true)`。这行代码极其关键，它会向正在执行人行查询的那个线程发送一个 `Interrupt` 信号。如果那个线程正在阻塞等待网络的响应，它会立刻抛出 `InterruptedException`，从而提前结束，**把宝贵的线程资源瞬间释放回线程池。**
* **不执着于完美的回滚：** 外部调用发出去的 HTTP 请求，其实很多时候是收不回来的（即使你取消了线程，远端的服务器可能还在算）。但在风控系统里，我们的态度是：**我不管远端算没算完，只要我本地判定你被拒绝了，我就立刻切断连接，释放我自己的本地资源，保护我自己的大盘。** ---

老周，这就是金融级网关里最硬核的“并发流控”。你今天在看项目组里风控编排的源码时，可以重点找找有没有类似 `CompletableFuture.anyOf` 或者手动控制 `Future` 状态的痕迹。他们可能包了一层极其复杂的自研中间件（比如封装好的 `TaskEngine`），但扒开表皮，底层的骨架一定是这套逻辑。

为了让你吃透这个概念，咱们先跳出代码。你想想你平时跑步的时候，如果突然感觉膝盖一阵剧痛，你会怎么做？你肯定会立刻停下来（断开），而不是咬牙继续猛跑，因为那样会导致半月板彻底撕裂（系统雪崩）。你会走一段路休息一下，过一会儿再小心翼翼地试着小跑两步（半开试探）；如果不再疼了，你再恢复正常的配速（闭合）。

**这就是当今微服务熔断器（如 Resilience4j）最核心的设计灵魂：状态机流转。**

我们把这个状态机代入到你调用的“人行征信接口”中，详细拆解它在底层到底是怎么运作的：

### 第一阶段：CLOSED（闭合状态）—— 岁月静好

* **场景：** 一切正常，客户来申请贷款，系统向人行接口发请求，正常返回结果。
* **底层动作：** 熔断器就像一个“透明的管道”，所有的请求都畅通无阻地发给外部服务器。但同时，它在暗中疯狂收集数据（比如记录最近 100 次请求里，有几次超时了，有几次报错了）。

### 第二阶段：OPEN（断开/熔断状态）—— 壮士断腕

* **场景：** 突然，人行的机房网络波动，接口开始大面积超时。
* **触发条件：** 熔断器发现，最近 100 次请求里，竟然有 50 次都超时或报错了（达到了你设置的 50% 失败率阈值）。
* **底层动作（最关键）：** 熔断器瞬间由 CLOSED 切换为 **OPEN**。
* **绝对防线：** 在接下来的一段时间内（比如你设置的 60 秒内），**任何指向人行接口的请求，连网络包都不会发出去！** 熔断器在 Java 内存层直接拦截，瞬间抛出 `CallNotPermittedException`，然后强行让代码走到你的 `Fallback`（兜底）方法里。
* **意义：** 外部接口已经瘫痪了，如果你还继续发请求，不仅外部接口恢复不了，你自己的 Tomcat 线程池也会因为全部卡在等待响应上而彻底耗尽。熔断就是为了保护建行自己的核心资源不受牵连。



### 第三阶段：HALF-OPEN（半开状态）—— 投石问路

* **场景：** 60 秒的冷却期过去了。人行的网络修好了吗？我们不知道。
* **底层动作：** 熔断器自动从 OPEN 切换为 **HALF-OPEN**。
* 这时候，它会变得极其谨慎。它只会偷偷放行**极其少量**的请求（比如只放 10 个请求过去）作为“敢死队”去探路，而其他的请求依然被拦截走 Fallback。
* **结果判定：** * 如果这 10 个探路请求又失败了 5 个，说明人行还没修好，状态机立刻打回 **OPEN**，继续冷却 60 秒。
* 如果这 10 个探路请求全部丝滑成功，说明外部故障彻底排除了，状态机华丽切回 **CLOSED**，全量恢复正常流量。

### 🛡️ Fallback（兜底）的业务哲学

熔断触发后，代码进入 `Fallback`，这时候前端客户会看到什么？这就是体现业务素养的地方了。

在零售风控里，如果查反欺诈接口被熔断了，我们绝对不能给客户弹出一个丑陋的 `500 System Error`。

* **优雅的 Fallback：** 返回一个特定的业务码（比如 `STATUS_MANUAL_REVIEW`），前端接收到后，给客户展示：“您的资料正在加速审核中，请稍后查看”。实际上，我们在后台把这笔单子悄悄转给了人工信审团队，或者丢进定时任务里，等接口恢复了再查一遍。

把这个状态机的流转机制装进脑子里，以后你再看到那些偶尔报错、一会又自动恢复的接口，就能一眼看穿它们底层的把戏了。
