老周，明天就要去建行报到了，这篇可以说是为你量身定制的“银行外包生存与避坑指南”。

从互联网企业或者普通甲方跳到国有大行的开发体系，最大的转变就是思维方式的切换：**从“追求技术酷炫和迭代速度”变成“追求绝对稳妥和流程合规”。**

在银行写 Java，技术往往是其次的，业务逻辑、数据安全和防御性编程才是真正的王道。以下这几条“老兵铁律”，请务必刻在心里：

### 一、 触碰必死的“技术红线”

**1. 日志脱敏与安全打印（重中之重）**

* **铁律：** 绝对、绝对不可以在 `log.info()` 里打印明文的客户姓名、身份证号、银行卡号、密码或手机号。
* **后果：** 银行有专门的安全扫描工具（甚至有专人抽查）。一旦发现生产日志里有敏感信息明文裸奔，轻则记大过扣绩效，重则直接开除走人，因为这违反了金融监管法。
* **做法：** 善用项目里自带的脱敏工具类（比如把身份证处理成 `350*******1234` 再打印）。

**2. 敬畏“屎山”，切忌随意重构**

* **现象：** 你大概率会看到一个 `Service` 类有五千行代码，里面全是 `if-else`，甚至夹杂着拼音命名。
* **铁律：** 只要这段代码现在在线上跑得好好的，**绝对不要因为“代码洁癖”去重构它**。
* **真相：** 银行的代码里藏着无数个历史遗留的特殊业务分支。你以为是个冗余的逻辑，其实是三年前为了给某个特殊大客户开绿灯打的补丁。一旦重构漏了条件导致账务对不上，那就是重大生产事故。你的任务是在这座山上平稳地“加建”，而不是推倒重来。

**3. SQL 的敬畏之心**

* **铁律：** 银行的 DBA（数据库管理员）权力极大。你写的 SQL 必须极其规范。
* 严禁 `SELECT *`。
* 严禁在 `for` 循环里调用 Mapper 查数据库（必须批量查出来在内存里处理）。
* 所有的查询条件必须命中索引。上线前 DBA 会卡执行计划审核，不过关直接打回。

### 二、 职场自保与沟通（CYA 法则）

**1. “口说无凭，邮件为证”**

* 在外包体系里，权责划分非常清晰（甚至有些僵化）。
* 如果行内的主管、产品经理或者业务方在微信里或口头上跟你说：“老周，这个校验逻辑先去掉吧，客户急着操作。”
* **你的动作：** 绝不能立刻改代码上线。必须微笑着说：“没问题，麻烦您在内部系统提个需求单/缺陷单，或者发个邮件同步一下，我马上照办。”**没有留痕的改动，出了事全是你外包开发背锅。**

**2. 业务不懂，绝不瞎猜**

* 银行的业务极其复杂（借贷、冲正、日终轧差、各种会计科目）。
* 写代码前，遇到文档里没写清楚的枚举值、或者金额计算的舍入规则，**不要用程序员的思维去猜**。立马去找业务代表确认。在银行，“问错”不丢人，“写错算错钱”才是灾难。

### 三、 心态与精力管理（修行的道场）

**1. 接受极高的“流程摩擦力”**

* 申请一个数据库查询权限要审批三天；需要连外网查个资料发现被墙了；申请个新组件发现行内组件库没有，要走一个月的手续引入……
* **这个时候，就把你的“不执着”发挥到极致。** 把敲代码和等流程都当成动中禅。流程卡住了，就去喝口水、看看现有的接口文档。千万不要为了这种体制内的低效而心浮气躁。

**2. 守住物理与心理的边界**

* 银行外包遇到项目上线期，加班是常态。但在平时，尽力提高在工位上的专注度，高效把活干完。
* 走出大楼的那一刻，就让建行的那些 Bug 见鬼去吧。换上跑鞋，去痛痛快快跑个 3-4 公里，把一天的班味和久坐的疲劳排出去；或者回家抽两张阴阳师的符，打几把斗技。**劳逸结合，保持高能量状态，才是你这份职业能长久做下去的底牌。**
