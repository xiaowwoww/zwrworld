老周，这个问题直接命中了分布式架构开发的核心命门！

在传统的单机数据库（如老版本的 Oracle 或 MySQL）时代，我们写 SQL 的心法是：“只要能查出来，越复杂越好”，几百行的表连接、子查询随便丢给数据库去算。

但在建行现在的 GoldenDB 或 GaussDB 这类分布式环境下，你必须强行扭转思维，建立**“节点意识”**。因为在你和真实数据之间，多了一层极其关键的 **Proxy（代理/计算）节点**。

为了不在生产环境写出把 Proxy 节点打挂的“夺命 SQL”，在分布式关系型数据库中开发查询语句，必须死守以下 **4 条军规**：

### 军规一：查询条件必须带上“分片键（Sharding Key）”

这是分布式查询的第一铁律，也是你昨天了解分片键后最直接的应用。

* **夺命写法：** `SELECT * FROM t_loan_order WHERE status = 'OVERDUE'`
* **后果（广播查询 Broadcast）：** 因为没有带分片键，Proxy 节点不知道逾期的订单到底在底层的哪台机器上。它只能硬着头皮把这条 SQL 发给底层的 **100 个物理节点**。100 个节点各自全表扫描，再把海量数据全部传回 Proxy 节点。Proxy 内存瞬间撑爆（OOM）。


* **架构师写法：** `SELECT * FROM t_loan_order WHERE customer_id = 'C123456' AND status = 'OVERDUE'`
* **原理（单点路由 Single-Node Routing）：** 因为带上了分片键 `customer_id`，Proxy 直接通过 Hash 算法算出这条数据在“节点 8”，于是只把 SQL 发给节点 8。性能瞬间提升百倍。



### 军规二：严禁大表的“跨库 JOIN”

在分布式库中，JOIN 是万恶之源。

* **场景：** 你要关联“借据表”（以借据号分片）和“客户表”（以客户号分片）。
* **夺命写法：** 直接写 `SELECT ... FROM 借据表 a JOIN 客户表 b ON a.客户号 = b.客户号`。
* **后果：** 因为这两张表的分片规则不同，相同客户的数据大概率**不在同一个物理节点上**。底层的数据库无法直接 JOIN，必须把两张表的数据全部拉到 Proxy 节点所在的内存里进行“笛卡尔积”运算，网卡和内存双双瘫痪。


* **架构师破局法：**
1. **数据亲和（Co-location）：** 在建表阶段，强行把两张表的分片键设置成一样的（比如都用客户号）。这样同一个客户的数据永远在同一个节点，可以直接用原生的 JOIN。
2. **Java 内存组装（最常用）：** 在代码里分两次查。先查出借据列表，提取出所有的客户号（`List<String> customerIds`），再用 `WHERE customer_id IN (...)` 去查客户表，最后用 Java 8 的 Stream API 在内存里把它们拼起来。



### 军规三：警惕“深分页（Deep Pagination）”陷阱

在前端页面做分页列表时，这是最容易踩的坑。

* **夺命写法：** `SELECT * FROM t_repayment_log WHERE type = 'NORMAL' LIMIT 1000000, 20`（查第 100 万条之后的 20 条）。
* **后果：** 在单机库里，这最多是查得慢点。但在分布式库里，Proxy 会要求底层的 10 个数据节点，**每个节点都查出前 1000020 条数据**，然后把这 1000 万条数据全部塞进 Proxy 节点的内存里进行重新全局排序，最后只截取 20 条给你。这绝对是灾难。
* **架构师写法（游标法 / Seek Method）：**
  绝对不能用单纯的 `OFFSET`。必须带上上一页最后一条数据的唯一标识（如主键 ID 或时间）。
  `SELECT * FROM t_repayment_log WHERE type = 'NORMAL' AND id > #{last_max_id} LIMIT 20`。

### 军规四：谨慎使用全局排序和聚合运算

* 当你在 SQL 里写 `ORDER BY`, `GROUP BY`, `MAX()`, `SUM()` 时，一定要意识到：**这些运算最终都是在 Proxy 节点的内存里完成的二次加工。**
* **建议：** 如果非要排序，必须确保筛选后的结果集非常小。能不排序的，尽量拉到 Java 业务层去排。用你的话说，不执着于让数据库去干所有的活，让 Java 帮它分担一些，劳逸结合。

老周，在分布式数据库里写 SQL，本质上就是一场**“如何不让 Proxy 节点过载”**的保卫战。你的每一次查询，都要在脑海里想象一下：这条 SQL 发出去，底层的 10 个节点会怎么跑？数据是怎么汇总回来的？

你在写代码时，遇到拿捏不准的 SQL，千万别直接上开发环境跑。
