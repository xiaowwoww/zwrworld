老周，这份对照表绝对是你明天入职第一天看表结构、写或者逆向生成实体类（Entity）时的“必备外挂”。

金融外包项目底层的数据库通常是重量级的 **Oracle**，或者是近几年做信创替代常用的 **MySQL / TiDB**。

我为你整理了这两种最主流数据库与 Java 数据类型的标准对应关系。尤其注意我在“业务场景”里标注的金融级用法：

### ☕ Java 与主流数据库字段类型标准对照表

| Java 数据类型 | MySQL 字段类型 | Oracle 字段类型 | 核心业务场景 / 避坑指南 |
| --- | --- | --- | --- |
| **`String`** | `VARCHAR`, `CHAR`, `TEXT` | `VARCHAR2`, `NVARCHAR2`, `CLOB` | 存姓名、流水号、证件号。**注意：** 身份证号、银行卡号千万别用数字型，必须用 `String` / `VARCHAR`。 |
| **`Integer`** | `INT`, `TINYINT` | `NUMBER(10)` | 存状态字典（如：0-处理中, 1-成功, 2-失败）、数量、年龄。 |
| **`Long`** | `BIGINT` | `NUMBER(19)` | **主键（PK）的绝对主力**。金融系统数据量极大，主键通常是雪花算法生成的 18-19 位数字，必须用 `Long` 和 `BIGINT`。 |
| **`BigDecimal`** | `DECIMAL(M,D)` | `NUMBER(P,S)` | **金融生命线。** 存账户余额、交易金额、利率、汇率。绝不能用 `Double`。 |
| **`java.time.LocalDateTime`** / `java.util.Date` | `DATETIME`, `TIMESTAMP` | `DATE`, `TIMESTAMP` | 存交易发生时间、创建/更新时间。**强烈建议：** 现在的规范项目都要求废弃老旧的 `Date`，改用 Java 8 的 `LocalDateTime`，它是线程安全的。 |
| **`java.time.LocalDate`** | `DATE` | `DATE` | 存不需要具体到秒的日期，比如：账期（2026-03-01）、客户生日。 |
| **`Boolean`** | `TINYINT(1)` | `NUMBER(1)` 或 `CHAR(1)` | 存逻辑开关（是/否）。**建行特色注意：** 很多老一代银行系统不用数字存布尔值，而是喜欢用 `CHAR(1)` 存 `'Y'` 和 `'N'`，遇到这种要在 Java 里做好转换映射。 |
| **`byte[]`** | `BLOB` | `BLOB` | 存二进制文件流，比如客户上传的身份证扫描件、电子签名图片。 |
| **`Double`** / `Float` | `DOUBLE`, `FLOAT` | `BINARY_DOUBLE` | 存非核心的科学计算数据、经纬度坐标、或者允许轻微误差的统计报表比例。**绝不允许碰钱。** |

### 💡 架构师的实战经验（入职第一天大概率用得上）

明天你拿到项目的建表 SQL 或者查库的时候，教你两个快速判断这个项目代码质量和年代的技巧：

1. **看时间类型：** 如果实体类里满屏的 `java.util.Date`，说明这大概率是个有年头的老项目；如果是 `LocalDateTime`，说明架构稍微新一点，或者经历过重构。
2. **看布尔值映射：** 如果用 MyBatis Plus，数据库是 `TINYINT(1)` 存 `0` 和 `1`，Java 里直接写 `Boolean`，框架会自动映射。但如果数据库存的是 `'Y'` 和 `'N'`，老代码可能会写死 `String`，高级一点的写法是实现一个 MyBatis 的 `TypeHandler` 做全局自动转换。
