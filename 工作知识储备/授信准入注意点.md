老周，分到“授信准入风控”这块，说明项目组把你当成能扛事的核心战斗力了！

在建行的零售贷款（比如“快贷”、“个人按揭”）体系里，计息引擎管的是“怎么赚钱”，而你负责的准入风控管的是**“怎么保命”**。你是守在资金大门前的第一道防线，决定了这笔钱能不能借、借给谁。

在这个极其敏感、逻辑极其密集的“重工业区”写代码，想要不背锅且游刃有余，这 **4 条风控老兵的实战铁律**你必须烂熟于心：

### 1. 斩断“同步等待”：敬畏外部接口的脆弱

准入风控最恶心的地方在于，你需要向无数个地方要数据：公安身份核验、人行征信系统、法院老赖失信库、行内黑名单、反欺诈引擎……

* **夺命写法：** 挨个串行调用。调公安等 2 秒，调人行等 3 秒。如果其中一个接口卡死了，你的系统线程池会瞬间被耗尽，整个建行 App 的贷款入口直接瘫痪。
* **架构师解法：并发编排与严格熔断。**
  必须把这些互不依赖的查询拆分开，使用 Java 的 `CompletableFuture` 进行异步并发调用。同时，所有外部 HTTP/RPC 调用必须强制加上**超时时间（Timeout）**和**熔断器（Circuit Breaker）**。如果非核心接口（比如某个外部的大数据信用分）超时了，直接执行降级逻辑（比如默认给个及格分或跳过），绝不能让它拖死主链路。

### 2. 戒掉“if-else”执念：做规则引擎的搬运工

准入规则是随国家政策和行内风险策略随时波动的。今天规定 18-60 岁能贷，明天可能就改成了 22-55 岁；今天有公积金就能过，明天可能还要看流水。

* **老兵铁律：绝对不要在 Java 代码里硬编码写 `if (age < 18 || inBlacklist) { return false; }`！**
* **正确姿势：** 建行内部绝对有自己封装好的一套**规则引擎（Rule Engine，底层可能是 Drools、LiteFlow 或自研引擎）**。
  你的职责不是去写判断逻辑，你的职责是“准备粮草”——把客户的年龄、收入、征信报告解析成一个个标准的 `Fact`（事实对象），然后一股脑扔给规则引擎，让引擎告诉你“通过”还是“拒绝”。把业务规则和代码彻底解耦，这是风控开发的基本修养。

### 3. 监管的达摩克利斯之剑：做到 100% “可回溯”

在风控系统里，“拒绝客户”是家常便饭，但你必须能回答出一个致命问题：**“凭什么拒绝他？”**

* 如果客户去银保监会投诉，或者行内审计部门来查账，你必须能拿出当时的铁证。
* **快照与留痕机制：** 客户点击申请那一瞬间，他填写的资料、外部接口返回的原始 JSON 报文（尤其是人行征信这种强监管数据），以及规则引擎最终命中的**“拒绝代码（Reject Code，比如 R001 代表命中欺诈黑名单）”**，必须作为“历史快照”原封不动地落库。哪怕第二天客户的征信变好了，你也有证据证明“当时拒他是有理有据的”。

### 4. 守住高压线：顶级敏感数据的脱敏与存储

你手里流转的，是客户最核心的隐私（身份证、人脸、征信报告、资产负债表）。

* **绝对红线：** 人行征信报告是金融业的最高机密。绝对不能在业务日志 `log.info()` 里明文打印征信明细！绝对不能把未经加密的敏感信息存入普通的 Redis 缓存中！一旦查出违规留存或泄露，这不是开除的问题，而是直接移交经侦。
* 在调试代码时，遇到要看敏感字段的值，用行内提供的脱敏工具类处理后再输出。
