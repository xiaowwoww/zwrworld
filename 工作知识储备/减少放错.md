在建行这种国家队里，**“不求有功，但求无过”**绝对不是一句贬义词，而是最高级的工程智慧。互联网公司讲究“小步快跑，试错迭代”，但在零售贷款这种动辄几亿资金流转的风控系统里，试错的代价就是职业生涯的终结。

既然你现在还在等审批入场，正是建立防御心智的最佳时机。我帮你总结了 **5 条绝对能保命的“防御性开发铁律”**，照着做，基本能帮你挡掉 99% 的黑锅：

### 1. 流程护体：拒绝一切“口头需求”与“微信承诺”

这是外包开发最容易踩的生死坑。

* **危险场景：** 行方的产品经理或者业务人员跑来你工位，或者在微信上跟你说：“老周，这个客户在网点等着放款呢，你赶紧把这行校验代码注释掉，先发个版让他过去。”
* **老兵铁律：永远只认系统里的单子（Jira / 缺陷单 / 需求文档）。**
* 任何没有落在纸面上的改动，最后出了生产事故，业务人员可以说“我没说过”，行方主管可以说“我没审批”，最后背锅被清退的只有你。
* **正确话术：** “没问题哥，您在系统里提个工单或者补发个邮件说明一下特殊情况，单子一流转到我这，我立马五分钟给您改好提 MR（合并请求）。”微笑着把皮球踢回流程里。



### 2. 代码隔离：敬畏“屎山”，只做增量，绝不做存量修改

建行的核心系统历史包袱极重。你可能会看到一个风控准入方法有两千行代码，里面不仅有拼音命名，还有各种魔数（Magic Number）。

* **危险动作：** 看到不爽的代码，凭借自己的技术洁癖，顺手给它重构了。
* **老兵铁律：严格遵守开闭原则（Open-Closed Principle）。**
* 那两千行代码之所以长那样，是因为它在过去五年里经历了无数次修修补补，里面藏着几百个你根本不知道的特殊业务分支。只要它现在线上没报错，**绝对不要动它**。
* 接到新需求，就在旁边老老实实地加一个 `if (newCondition) { // 你的新逻辑 }`，或者单独抽一个新方法。改动范围越小，你引入新 Bug 的概率就越无限趋近于零。



### 3. 边界防御：永远把前端和上游系统当成“骗子”

在风控系统里，绝对不能信任任何人传给你的数据。

* **老兵铁律：入参的极致校验。**
* **空指针（NPE）防范：** 银行接口的 JSON 报文极其庞大。在组装风控 Fact 时，获取深层嵌套对象必须步步为营。多用 `Optional.ofNullable()`，严禁直接 `a.getB().getC()`。
* **类型与范围校验：** 哪怕前端发誓他们已经做了拦截，你后端也必须用 `@Valid` 和 `@NotBlank`、`@Max` 等注解再拦一次。年龄传了负数？身份证传了 15 位？直接在 Controller 层抛出异常打回去。



### 4. 异常收口：宁可“误杀”转人工，绝不“漏网”放资金

这是咱们昨天聊过的，但在开发实操中必须要落地。

* **危险动作：** 调用外部反欺诈接口，为了不让程序报错，顺手写个 `catch (Exception e) { return true; }` 把异常吞了，让流程继续走。
* **老兵铁律：风控引擎的 Fail-Closed（安全断电）法则。**
* 在风控准入里，遇到任何拿不准的异常（空指针、网络超时、数据库锁超时），**一律向外抛出明确的业务异常，或者将状态置为“转人工复核”**。宁可让客户多等半天，也绝对不能让带有异常状态的单子溜进下一个放款环节。



### 5. 留痕保命：打印“完美的案发现场”

排查 Bug 时，最怕的是不知道当时发生了什么。

* **老兵铁律：关键节点的出入参必须打印日志（注意脱敏）。**
* 收到请求的第一时间：打印唯一的 `TraceID` 和入参。
* 调用外部征信接口前：打印请求报文。
* 外部接口返回后：打印响应报文和耗时（便于日后扯皮时证明是对方接口慢）。
* 规则引擎返回结果后：打印命中的规则代码（Reject Code）。
  有了这些日志，日后前端、测试或者业务方来找你质问“为什么系统有问题”，你直接甩出日志链路，一秒钟自证清白。
