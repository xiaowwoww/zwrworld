### 1. 告别 `if-else` 的终极形态：多产品线的“策略路由”

零售贷款不是一个产品，它下面挂着几十个子产品：“快贷”、“车贷”、“装修贷”、“农户贷”。

* **初级开发的灾难现场：** 在风控准入的入口，写出这种满屏代码：
  `if ("KUAIDAI".equals(productCode)) { // 走快贷准入逻辑 } else if ("CHEDAI".equals(productCode)) { // 走车贷逻辑 }`
  只要新上一个贷款产品，就得改这坨屎山，极其容易把老产品改出 Bug。
* **老兵打法（策略模式 + 工厂模式）：**
  在 Spring Boot 里，彻底消灭业务分支路由。
  定义一个通用的 `RiskAdmissionStrategy` 接口。然后各个产品线自己去实现，打上类似于 `@Service("riskStrategy_KUAIDAI")` 的注解。
  在网关入口处，直接用一个 `Map<String, RiskAdmissionStrategy>` 注入所有的策略实现类。请求来了，直接 `map.get(productCode).execute(fact)`。新增产品，只需要新增一个类，主流程一行代码都不用改。

### 2. 特征计算的绝对禁忌：永远不要在代码里写 `new Date()`

在组装 Fact（风控特征）的时候，我们需要算很多时间维度的数据，比如“客户当前的年龄”、“距离上次逾期的天数”。

* **夺命陷阱：** 很多开发会顺手写 `int age = currentYear - birthYear;`，这里的 `currentYear` 是用系统当前时间 `new Date()` 算出来的。
* **灾难后果：** 风控系统有一个极其重要的功能叫**“历史案件回放（Replay）”**。审计部门在 2026 年，要把 2024 年的一笔拒件重新跑一遍，看看当时的规则有没有问题。如果你代码里写了 `new Date()`，客户的年龄凭空老了两岁！原本符合条件的，回放时变成了不符合。
* **老兵铁律：一切时间特征，必须以“客户点击申请那一瞬间的交易时间（Apply_Time）”为锚点！** 严禁在任何风控特征计算逻辑中获取服务器的当前物理时间。

### 3. 宁可错杀，绝不放过：灾难降级的“Fail-Closed”原则

昨天我们聊了熔断和降级，但在银行业，风控系统的降级有着极其冷酷的底线。

* **互联网思维（Fail-Open）：** 如果淘宝的推荐引擎挂了，为了不影响用户下单，就默认展示热门商品（放行）。
* **金融风控思维（Fail-Closed）：** 如果建行的规则引擎彻底宕机了，或者人行征信的专线被挖断了，你的准入接口该怎么返回？
  **绝对不能因为“系统异常”就默认给客户放行！** 一旦放进去一堆黑户，就是极其严重的金融事故。
* **代码处理：** 在最外层的 `catch (Exception e)` 里，只要遇到拿不准的全局异常，一律返回 **“系统审核中，转人工处理（Pending）”** 或者明确提示 **“系统维护中，请稍后再试”**。守住资金大门，不妄动，是风控开发的最高准则。

### 4. 降维打击：应对团伙欺诈的“图数据库”思维

普通的黑名单防不住专业的“黑产羊毛党”。他们会用干净的身份证、新手机号来申请贷款。

* **隐秘的关联：** A 是正常客户，B 是骗子。传统数据库很难查出“A 和 B 虽然不认识，但他们昨天连过同一个城中村的 WiFi，且 A 曾给 B 的某个远房亲戚转过账”。
* **架构演进：** 建行内部一定会有一套极其强大的**知识图谱（Graph Database，比如 Neo4j、HugeGraph 等）**系统。
  作为 Java 开发，你不需要懂底层的图遍历算法，但你必须建立这种思维。在调用反欺诈服务时，你传过去的不仅仅是客户的身份证，可能还要收集并传入设备的 MAC 地址、IP 地址、GPS 定位。图数据库会在底层做几度人脉的“碰撞交叉”，一旦发现这个客户在图谱里和某个“黑产节点”距离过近，会直接返回极高的风险评分。
