### 1. 拦截海量黑产的终极武器：布隆过滤器（Bloom Filter）

在准入风控的最外层，有一项极其耗时的操作：**查黑名单**。
建行积累的各类黑灰产名单、老赖名单、欺诈设备 IP 库，数据量是亿级别的。

* **危险动作：** 客户一进件，你就拿着他的身份证号去数据库里 `SELECT count(*)`，或者把几千万条黑名单全塞进 Redis 里。前者会把数据库查死，后者会把 Redis 内存撑爆。
* **架构老兵打法：** 在网关或风控最前置的拦截器里，引入 **布隆过滤器（Bloom Filter）**。
  它极其轻量，占用内存极小。当请求过来时，先过一遍布隆过滤器：
* 如果它说“**不在**黑名单里”，那这个人绝对清白，直接放行进入后续流程。
* 如果它说“**可能在**黑名单里”，你再去查真正的 Redis 或数据库进行二次确认。
  这能帮你挡掉 99% 的无效数据库查询，极其优雅。



### 2. 线程池的“舱壁隔离”模式（Bulkhead Pattern）

前天咱们聊了用 `CompletableFuture` 做并发编排，但这其实还不够。

* **恐怖场景：** 假设你有一个全局的“风控专用线程池”。今天人行征信接口突然卡顿，所有的线程都在等人行返回。这时候，另一个本来只需要查“行内极速黑名单”的极速请求进来了，却发现线程池已经被耗尽了，只能跟着排队报错。
* **老兵防线：** 就像打斗技排兵布阵一样，你不能把所有核心资源都绑在一起，一旦一个被控，全盘皆输。必须做**舱壁隔离**：
* `人行征信专属线程池`（核心线程 20，最大 50）
* `公安反欺诈专属线程池`（核心线程 10，最大 20）
* `行内规则专属线程池`
  这样，就算人行征信彻底宕机，把它的专属线程池撑爆了，也绝对不会影响行内其他规则的正常运算。



### 3. 合规底线：征信授权书的“强校验”与“存证”

这是很多从互联网跳进金融圈的开发最容易忽略的死穴。

* **业务红线：** 没有客户的明确授权，银行系统**绝对不能**私自去查他的人行征信。如果查了，一告一个准。
* **开发职责：** 在你调用人行接口查征信的 Java 代码之前，必须有一个前置校验步骤：**校验该客户的“电子征信授权书”是否已经签署，并且 PDF 存证是否已经落盘并加上了电子签章。**
* **修心哲学：** 面对这种繁琐的校验流程，保持不急不躁的定力。把它当成业务流转中的一个既定因果条件，前置条件不满足，直接 `return` 阻断，不强求流程推进。

### 4. 生产排雷神器：请求报文的“防篡改哈希（Hash）”

风控接口一旦报错，业务方经常会找过来说：“明明这个客户条件很好，为什么系统拒了他？”

* **隐藏的坑：** 你去查日志，发现当初传进来的参数确实是不符合条件的。但前端小哥打死不承认，说是你后端把数据接收错了。
* **防背锅打法：** 在入参的最外层，让前端把所有的参数拼接起来，用 SM3 或 SHA-256 算法生成一个 `Signature（签名）` 一起传过来。
  你的风控接口接客的第一步，就是自己算一遍签名，比对一致才往下走。日后如果扯皮，你直接把这串不可逆的哈希值甩出来：“报文在传输过程中就是这样的，跟我后端的处理逻辑毫无关系。”
